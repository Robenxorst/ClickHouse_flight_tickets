# ClickHouse Flight Tickets
Создание витрины данных на кластере ClickHouse для расчета метрик авиаперелетов.

## Задание
Подготовить данные из таблиц-источников таким образом, чтобы их было удобно испопользовать для быстрого расчёта бизнес-метрик, создания BI-дашбордов, ad-hoc аналитики.
Ключевые бизнес метрики, которые хотим рассчитать:
1. **ASK(Available Seat Kilometer)** - "доступные кресло-километры",  важная метрика авиаперелётов, показатель "ёмкости" рейса. 
2. **RASK (Revenue per Available Seat Kilometer)** — удельная выручка на кресло-километр. Она показывает, насколько эффективно используются имеющиеся "объёмы авиаперевозок". То есть показывает общую эффективность эксплуатации самолетов.
  ```
  RASK = Revenue / ASK
  ```
3. **Yield (доходность)** - это тоже "нормированная" выручка, как и RASK, но уже на пассажиро-километр. Пассажиро-километр — **RPK (revenue passenger-kilometer)** похожа на ASK, но учитываются только физически занятые платящими пассажирами места, а не просто доступные кресла (и не просто проданные билеты).
   ```
   Yield = Revenue / RPK
   ```
4. **Load Factor (загруженность, заполняемость)** - коэффициент, который показывает насколько хорошо заняты места, т.е. не летают ли самолеты пустыми. Является отношением сумм RPK и ASK:
   ```
   Load Factor = RPK / ASK
   ```
## Исходные данные

В качестве исходных данных у нас будут:

1. Две крохотные таблицы: `airports` (аэропорты) и `aircrafts` (модели самолётов). Их мы превратим в словари, чтобы не тратить ресурсы на джойны.
2. Две таблицы побольше, в которых будут авиаперелёты. Одну из них, `flights`, мы будем считать append-only (только добавляются новые данные, старые в норме не редактируются, стандартный сценарий для ClickHouse). Вторая, `flight_statuses`, содержит данные, которые могут обновляться: статус рейса, фактическое время прибытия и т.д. Эти две таблицы реплицируем, но не шардируем.
3. Две таблицы, которые мы будем считать "большими". Сейчас в них по несколько миллионов строк, но будем считать, что они сильно вырастут, и стоит заранее разбить каждую по шардам. Это `ticket_flights` (данные о билетах) и `boarding_passes` (посадочные талоны; данные о пассажирах, которые непосредственно сели в самолёт). Не все люди, купившие билет, действительно придут на рейс.

Будем считать, что новые данные приходят в определённом порядке. Общая информация о рейсе появляется задолго до того, как купят первые билеты на него. В свою очередь, посадка в самолёт происходит после того, как все купили билеты на этот рейс. А вот статус рейса может обновиться в любое время.

## Общая концепция
Выстроим ступенчатую систему обработки данных. Будем использовать материализованые представления (materialized view) - триггер, который срабатывает на вставку данных в таблицу-источник и по заданным правилам добавляет данные в другую, целевую таблицу.

Этапы выполнения задачи:
1. Создание двух словарей `aircrafts_dict` и `airports_dict`.
2. Создание таблиц `flights_raw`, `flights` и материализованного представления `flights_mv` для обновления данных во flights + отдельную таблицу `flight_statuses`, где может динамично меняться статус. Все это выполняется на реплицированном кластере(4 дублирующихся сервера).
3. Создание таблицы с сырыми данными `ticket_flights_raw_local` и Distributed-таблицы `ticket_flights_raw` на шардированом кластере (2x2 - два шарда по две реплики на каждом).
4. Создание широкой таблицы `ticket_flights_wide_local` на шардированном кластере кластере, объединяющая все столбцы из  `ticket_flights_raw` и `flights`, Distributed-таблицы `ticket_flights_wide` и материализованное представление `ticket_flights_raw_to_wide_mv`.
5. Создание по аналогии таблиц `boarding_passes_raw_local` , `boarding_passes_raw` , `boarding_passes_wide_local` , `boarding_passes_wide` и `boarding_passes_raw_to_wide_mv`.
6. Создание агрегирующей таблицы с метриками `flight_metrics_local` на шардирующем кластере c внутренней **проекцией** `metrics_aircraft_code_projection`( помогает выдавать агрегированные (просуммированные) значения всех 4-х наших сырых метрик в группировке по дате и модели самолёта) +  Distributed-таблицы `flight_metrics`.
7. Создание трех матераилизованных представлений `flights_to_metrics_mv` , `tickets_to_metrics_mv` и `boarding_to_metrics_mv`, которые пишут в одну и ту же агрегрующую таблицу `flight_metrics_local`, притягивая данные из разных таблиц.

## Запрос для расчёта ratio-метрик

В качетсве итога подготовим SELECT запрос к агрегирующей таблице, который будет выдавать основные метрики маршрутов в удобном для бизнес-пользователей формате. Используемые столбцы:
1. `route` — маршрут, два аэропорта через черту;
2. `dep_city` — город аэропорта вылета;
3. `arr_city` — город аэропорт назначения;
4. `total_ask` — сумма ASK. Помимо использования для расчёта других метрик, показывает, насколько "большой" это маршрут. На уровне конкретного полёта ASK отражает, насколько вместительный самолёт и насколько далёкий маршрут.
5. `total_passengers` — сколько всего пассажиров по факту перевезли
6. `total_revenue` — суммарная выручка
7. `yield` — метрика Yield (доходность), рублей за пассажиро-километр, во сколько пассажиру обходится километр перелёта
8. `load_factor` — метрика Load Factor (загрузка), насколько заполнены самолёты
9. `rask` — метрика RASK, рублей за кресло-километр, насколько эффективно используются самолёты
10. `performance` — поле, отражающее в каком квадранте по Yield и Load Factor находится рейс. Принимает значения: 'Healthy', 'Demand Issue', 'Pricing Issue', 'Poor'.В качестве порогов используем такие значения: считаем, что минимальный "хороший" Yield это 11 рублей за пассажиро-километр. Для загруженности считаем, что "хорошая" загруженность самолёта это хотя бы 70% занятых кресел.


Используем только маршруты, у которых получился total_revenue больше 0. 

Сортируем запрос, показывая те маршруты, которые более всего отстают от "целевого" RASK. Чем сильнее отставание от целевого RASK, тем выше должен быть маршрут. При этом чем "больше" маршрут, тем выше он должен быть тоже. Считаем целевым RASK для всех маршрутов 70% * 11 = 7.7 рублей за пассажиро-кресло (граница квадрантов performance).
